# GATEWAY-ALERT-ROUTING-001: Alert Routing and Agent Notifications

**Status:** v1 draft
**Priority:** P1
**Owner:** @pulse
**Date:** 2026-04-26
**Related:** GATEWAY-SERVICE-ACCOUNTS-001, GATEWAY-ACTIVITY-VISIBILITY-001, GATEWAY-PASS-THROUGH-MAILBOX-001

## Why This Exists

Gateway alerts should not be anonymous noise. When a service account, task,
automation, or operator action creates an alert, the user needs to know:

- where the alert came from;
- which agent is responsible for it;
- whether that agent was notified live, queued in an inbox, or assigned a task;
- where to change the routing next time.

This is P1 UX because alerts are one of the main places the system either feels
connected or feels like a pile of disconnected messages.

The product rule:

> Alerts are routable work. Every alert has a source, a target, and a visible
> delivery state.

## Concepts

### Audit Export Sink

Some alert sources are not generated by a human workflow. They come from
security and observability systems that need the Gateway/aX registry to be an
auditable control point.

Named first targets:

- Splunk
- Wiz

The connector abstraction should be generic enough for other SIEM, CNAPP, and
log-routing systems. These sinks should receive structured Gateway/aX events,
not scraped UI text.

Minimum event families:

- Gateway login/session created, refreshed, revoked, or failed;
- fingerprint approval requested, approved, rejected, drifted, or revoked;
- agent credential minted, rotated, used, or denied;
- service-account message sent;
- agent-authored action performed through Gateway;
- suspicious direct-token or wrong-identity usage detected.

These exports must preserve the actor chain: bootstrap user, Gateway session,
service account if present, acting agent, target agent/user, space, and
fingerprint/ref identifiers. This lets security tooling answer "which Gateway
on which machine caused this action?" without exposing raw tokens.

### Alert Source

An alert source is usually a service account:

- `@notifications`
- `@security-alerts`
- `@logs`
- `@reminders`
- `@switchboard-<space>`

Sources are not live agents. They can send, schedule, and route messages, but
they do not think or reply. See **GATEWAY-SERVICE-ACCOUNTS-001**.

### Notification Target

A notification target is the agent, user, or group that should receive future
alerts from a source or alert class.

Examples:

- security alerts -> `@backend_sentinel`
- deployment failures -> `@orion`
- local Gateway issues -> `@codex-doorbell`
- task reminders -> assigned task owner

The target may be:

- a live/channel agent;
- a pass-through mailbox agent;
- a user;
- a future group/list.

### Alert Class

An alert class names the category, for example:

- `security`
- `logs`
- `deployment`
- `gateway`
- `reminder`
- `task`
- `qa`

The class lets Gateway route recurring alerts without the user selecting a
target every time.

## UX Contract

### Settings

Gateway settings should expose an **Alerts** section where the user can choose
default targets:

| Setting | Example |
| --- | --- |
| Default alert target | `@codex-doorbell` |
| Security alerts | `@backend_sentinel` |
| Deployment alerts | `@orion` |
| Reminder alerts | `@pulse-cc` |
| Task assignment notifications | assigned agent |

The settings UI should show the sender/source account and the target separately.
For example:

```text
Security alerts
From @security-alerts
Send to @backend_sentinel
```

### Share Icon

When a user shares or forwards an alert to an agent, Gateway may offer:

- `Share once`
- `Route future alerts like this to @agent`

The second option creates or updates an alert-routing rule for that alert class
or source. This should feel like teaching the system where future work belongs,
not like configuring a complex automation engine.

### Activity Stream

The activity stream must show:

- alert source;
- routed target;
- delivery state;
- task assignment state when applicable.

Examples:

```text
Security alert routed
From @security-alerts to @backend_sentinel
Delivered live
```

```text
Gateway alert queued
From @switchboard-ed81ae98 to @codex-doorbell
Delivered to inbox
```

```text
Reminder assigned
From @reminders to @pulse-cc
Task assigned
```

Do not show fake live processing for service accounts. If the target is
pass-through, the activity should say it landed in that agent's inbox. If the
target is live, subsequent agent-processing events attach to the same alert
message when backend support allows.

## Data Model Sketch

```json
{
  "alert_route_id": "route_gateway_default",
  "space_id": "49afd277-78d2-4a32-9858-3594cda684af",
  "source_agent_id": "service_security_alerts",
  "source_name": "security-alerts",
  "alert_class": "security",
  "target_type": "agent",
  "target_agent_id": "backend_sentinel_id",
  "target_name": "backend_sentinel",
  "delivery_mode": "auto",
  "created_by": "user",
  "updated_at": "2026-04-26T00:00:00Z"
}
```

Routing should support both source-specific and class-specific rules. Source
rules win over class rules; class rules win over the Gateway default.

## CLI Shape

```bash
ax gateway alerts routes list
ax gateway alerts routes set --class security --to backend_sentinel
ax gateway alerts routes set --source security-alerts --to backend_sentinel
ax gateway alerts routes set --default codex-doorbell
ax gateway alerts send --source security-alerts --class security "Suspicious token use detected"
```

The existing `ax alerts` surface may remain, but Gateway-owned alert routing
should be available under `ax gateway` because the Gateway dashboard owns the
operator UX.

## Task Integration

Tasks are alerts with durable ownership. When an alert creates or updates a task:

- the task owner is the notification target unless explicitly overridden;
- the task activity references the original alert message;
- future task notifications route to the assigned owner;
- if the task owner changes, future task notifications follow the new owner.

The task assignment notification should not create a second unrelated alert
thread unless the user explicitly shares it elsewhere.

## Acceptance Tests

- User can create or select a service account as an alert source.
- User can set a default alert target in Gateway settings.
- User can set an alert-class target, such as `security -> @backend_sentinel`.
- Sharing an alert to an agent can optionally make that route the future default
  for the alert source/class.
- Sending an alert records source, target, and delivery state in the activity
  stream.
- Sending an alert to a pass-through agent increments that agent's mailbox badge
  and says `Delivered to inbox` / `Queued in Gateway`, not `Working`.
- Sending an alert to a live agent shows live processing if the agent starts
  work.
- Task-created alerts route future task notifications to the assigned agent.
- Removing or disabling a target leaves the route visible as broken and prompts
  the user to pick a replacement instead of silently dropping alerts.

## Open Questions

- Should alert routes live entirely in Gateway local state first, or should the
  backend own them so they follow the user across devices?
- Should `Share once` / `Route future alerts` be backend message actions, Gateway
  drawer actions, or both?
- How should group targets be represented before we have first-class agent
  groups?
